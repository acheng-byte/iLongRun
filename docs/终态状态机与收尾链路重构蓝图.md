# iLongRun 终态状态机与收尾链路重构蓝图

> 本文是基于真实 coding 跑测样本抽象出的**通用内核整改说明**，不是针对某个业务任务的特判适配。

## 1. 目标

本轮重构只做一件事：

> 让新 run 的账本、review / audit / adjudication / finalize、以及终态摘要形成**唯一可信真值**。

从 `v0.9.0` 开始：

- 不再兼容旧 run 账本
- 不再接受 top-level `scheduler.status`
- 不再允许 `blocked`/`failed` 复用 `COMPLETION.md`
- 不再允许 `verification` 与 run 终态表达互相冲突

## 2. 新 canonical 状态机

`/Users/zscc.in/Desktop/AI/ilongrun-V2/ilongrun-repo/scripts/_ilongrun_lib.py`

新 run 只允许四种主状态：

- `running`
- `blocked`
- `completed`
- `failed`

其中：

- `scheduler.json.state` 是**唯一** run 主状态真值
- 旧的 top-level `status` 已废弃，出现即判错
- `verification.state` 不再独立决定终态，只能作为诊断辅助

## 3. 新收尾链路

唯一允许的收尾顺序：

```text
review -> audit -> adjudication -> finalize
```

约束如下：

1. `final-review.md` 是 audit 主结论真值
2. `adjudication.md` 只能在有效 `final-review.md` 之后生成
3. adjudication 只允许两个决策：
   - `proceed-to-finalize`
   - `return-for-fix`
4. 只有 `proceed-to-finalize` 才允许进入 `completed`
5. `return-for-fix` 必须把 run 置为 `blocked`

## 4. 终态摘要文档语义

新规则下，终态摘要完全分离：

- `COMPLETION.md`：只允许 `completed`
- `BLOCKED.md`：只允许 `blocked`
- `FAILED.md`：只允许 `failed`

因此以下情况都会被视为错误：

- `blocked` run 仍写 `COMPLETION.md`
- `failed` run 仍写 `COMPLETION.md`
- `completed` run 同时保留 `BLOCKED.md` / `FAILED.md`
- `running` run 提前出现任意终态摘要

## 5. finalize 规则

`completed` 的前置条件：

- `claimVerification` 已 complete
- `reviews/final-review.md` 存在且 verdict 可解析
- `reviews/adjudication.md` 存在且 decision=`proceed-to-finalize`
- review gates 全部 complete
- `pendingMustFixCount = 0`
- 本地 verify 的 finalize precheck 通过

`blocked` / `failed` 的规则：

- 必须写入对应终态摘要
- `verification.state` 不能保持 `passed`
- 默认不清理 `active-run-id`
- detached 服务是否仍在运行，只作为 runtime residue 记录，不代表成功完成

## 6. workstream done 的新约束

任意 workstream 写成 `done/complete` 前，至少满足：

- `result.md` 非空且非 placeholder
- `evidence.md` 非空且非 placeholder
- acceptance / verify checklist 达到阈值

额外约束：

- `define / plan` 仍保留 `result/evidence`
- 但完成时由系统自动写入真实结构化摘要，禁止继续保留 `Pending result/evidence` 模板

## 7. verify / doctor / status 的对齐原则

### verify
- 对 `scheduler.status`、终态摘要错配、review/adjudication 时序错误直接 hard-fail
- `blocked / failed` 会被强制降级 verdict，不再出现乐观 completion score

### doctor
- 检查 canonical state 冲突
- 检查 terminal summary 缺件或错配
- 检查 placeholder-done
- 检查工作区污染

### status board
- 明确展示 `COMPLETION.md / BLOCKED.md / FAILED.md`
- 只在 `completed + completion 存在 + active 已清` 时显示“已稳定完成”

## 8. 为什么不兼容旧 run

这次重构的目标是**让新 run 正确，而不是平滑兼容历史歧义**。

历史 run 的问题主要在于：

- `state` / `status` 双真值
- completion / blocked 文档混用
- review / adjudication 时序不可信
- 依赖 reconcile 才能“看起来正确”

因此 `v0.9.0` 以后：

- 旧 run 不再保证可读
- 旧 run 只用于人工复盘
- 新 run 一律按新 schema 写盘

## 10. 对后续跑测的建议

新的样例跑测重点看这几项：

1. `scheduler.state` 是否始终是唯一主状态
2. `completed / blocked / failed` 是否只生成匹配的终态摘要
3. `final-review` 与 `adjudication` 的生成时机是否严格正确
4. define/plan 的 done 是否还会残留 placeholder
5. `active-run-id` 是否只在真正 completed 时清理
