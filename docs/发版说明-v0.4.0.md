# 发版说明 v0.4.0 — run 协议统一、模型配置化、旧插件治理

> 发版日期：2026-04-07

## 概述

v0.4.0 聚焦三个长期痛点：

1. 同一 run-id 同时写入 `.copilot-ilongrun/<run-id>/` 和 `.copilot-ilongrun/runs/<run-id>/`，造成 verify / finalize 读错账本，出现“状态漂移 / 自动中断”。
2. 旧 `copilot-mission-control` 插件仍 enabled 时，工作区会被 hooks 自动写出 `.copilot-mission-control/`，和 iLongRun 并存两套状态根。
3. 默认模型分散在多个脚本里，用户难以统一修改。

这一版的目标就是：**把 run 目录、工作区状态根、默认模型、detached 启动看板全部统一起来。**

## 这次改了什么

### 1) canonical run 目录统一

现在唯一合法的 run 路径是：

```text
.copilot-ilongrun/runs/<run-id>/
```

旧的：

```text
.copilot-ilongrun/<run-id>/
```

不再作为真值目录。

当 helper 检测到同一个 run-id 同时存在两套目录时，会自动：

- 合并 legacy run 中的 review / result / evidence / journal / hook events
- 优先保留更有信息量的 scheduler 状态
- 写入迁移记录到 `.copilot-ilongrun/legacy-imports/run-merges/`
- 最后删除旧根目录 run，只保留 canonical run

### 2) 旧 `copilot-mission-control` 自动治理

从 v0.4.0 开始：

- `install.sh` 会先尝试执行：
  ```bash
  copilot plugin uninstall copilot-mission-control
  ```
- `ilongrun-doctor` 会再次检查旧插件是否仍启用
- `copilot-ilongrun` 在真正开跑前也会做最后一次兜底检查；若自动卸载失败，会直接拒绝开跑，而不是继续污染工作区

如果当前工作区已经存在 `.copilot-mission-control/`：

- 空壳或仅 hook 日志：归档后删除
- 含 legacy 内容：归档到 `.copilot-ilongrun/legacy-imports/` 后删除

最终目标是：**工作区只保留一个 `.copilot-ilongrun/` 状态根。**

### 3) 默认模型改为 JSONC 单一配置

新的主配置路径：

```bash
~/.copilot-ilongrun/config/model-policy.jsonc
```

仓库默认模板：

```bash
./config/model-policy.jsonc
```

默认映射：

- `ilongrun` → `claude-sonnet-4.6`
- `ilongrun-status` → `claude-sonnet-4.6`
- `ilongrun-prompt` → `claude-sonnet-4.6`
- `ilongrun-resume` → `claude-sonnet-4.6`
- `ilongrun-doctor` → `claude-sonnet-4.6`
- `ilongrun-coding` → `claude-opus-4.6`
- coding 最终终审 → `gpt-5.4`

配置优先级：

```text
--model > commandDefaults > skillDefaults > roleModels > fallback
```

此外：

- JSONC 注释区直接列出了可复制 slug
- README 会说明哪些 slug 是“本机已验证”、哪些只是“官方支持、建议先 doctor/probe”
- 通知链路依然纯本地，不引入 notificationModel

### 4) detached 启动看板统一

现在 `ilongrun` / `ilongrun-coding` / `ilongrun-resume` detached 启动成功后，不再只输出三行路径，而是统一输出中文品牌化看板，包含：

- run id
- 当前状态
- 当前阶段
- 运行模式
- 任务画像
- 执行模型
- 最终终审模型
- 工作区 / log / meta 路径
- `ilongrun-status <run-id>` / `ilongrun-resume <run-id>` 快捷命令
- 下一步建议

其中 `ilongrun-coding` 的启动看板已和 `ilongrun-status` 的术语风格保持一致。

## 升级建议

```bash
curl -fsSL https://raw.githubusercontent.com/izscc/iLongRun/main/install.sh | bash
ilongrun-doctor --refresh-model-cache
```

如果你之前遇到过：

- `.copilot-ilongrun` / `.copilot-mission-control` 同时存在
- `.copilot-ilongrun/<run-id>/` 和 `.copilot-ilongrun/runs/<run-id>/` 同时存在
- `ilongrun-coding` detached 输出过于简陋

建议在目标工作区执行一次：

```bash
ilongrun-doctor
```

它会顺手做冲突治理与旧状态清理。
