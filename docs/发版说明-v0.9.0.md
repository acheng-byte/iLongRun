# iLongRun v0.9.0 发版说明

## 概述

`v0.9.0` 是一次**破坏性账本重构**。

这次发版不再尝试兼容历史 run，而是直接把新 run 切到新的 canonical 终态状态机，重点解决：

- run 终态语义冲突
- `review / audit / adjudication / finalize` 时序不可信
- `blocked` / `failed` 误写 `COMPLETION.md`
- `verification` 与 run 终态打架
- define/plan 的伪完成占位产物

## 新增

- 新增 `BLOCKED.md`：仅 `blocked` run 生成
- 新增 `FAILED.md`：仅 `failed` run 生成
- 新增 `docs/终态状态机与收尾链路重构蓝图.md`

## 增强

### 账本真值
- `scheduler.json.state` 成为唯一 run 主状态真值
- 新 run 只允许 `running / blocked / completed / failed`
- top-level `scheduler.status` 现在属于非法字段

### 收尾链路
- `review -> audit -> adjudication -> finalize` 改为严格单向链路
- `adjudication.md` 只能在有效 `final-review.md` 之后生成
- adjudication 只允许：`proceed-to-finalize` / `return-for-fix`
- `claimVerification` 未完成时，`completed` 直接阻断

### 终态摘要
- `COMPLETION.md` 只允许 `completed`
- `BLOCKED.md` 只允许 `blocked`
- `FAILED.md` 只允许 `failed`
- `running` run 若提前出现终态摘要，会被 verify / doctor 直接报错

### workstream 规则
- workstream done 若缺 result/evidence，直接 hard-fail
- define/plan 完成时自动写入真实结构化 result/evidence，避免 placeholder 混入 done

### 诊断与看板
- status board 识别三类终态摘要
- doctor 增加 terminal summary 错配检查
- blocked/failed 的通知默认打开匹配摘要，而不是 `COMPLETION.md`
