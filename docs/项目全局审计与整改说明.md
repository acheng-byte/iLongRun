# 项目全局审计与整改说明

本次审计目标：检查 iLongRun 在连续多版本迭代后，是否存在逻辑漂移、命名残留、版本说明不同步、README 对新手不友好、以及 `--model` 语义与实际实现不一致的问题。

## P1 必修问题（已整改）

### 1. `--model` 只是入口优先，不是全链路强制

**现象**

- 顶层 `selectedModel` 可以被显式指定
- 但 `codingAuditModel`、`mission.modelAllocation`、review/audit 路径仍可能继续沿用默认强模型

**风险**

- 用户以为自己全程在用某个模型
- 实际终审或后续阶段可能偷偷切到其他模型
- 额度控制与结果可解释性都被破坏

**整改**

- 显式 `--model` 改为全链路强制语义
- review / audit / finalize 全部沿用同一模型
- 禁止跨模型 fallback，只允许同模型重试

### 2. 终审相关命名把模型写死

**现象**

- `ilongrun-gpt54-audit-reviewer.agent.md`
- `reviews/gpt54-final-review.md`

**风险**

- 与当前可配置 / 可强制指定模型的事实冲突
- 用户会误以为 final audit 永远只能是 GPT-5.4

**整改**

- agent 改为 `ilongrun-final-audit-reviewer.agent.md`
- 审查报告路径改为 `reviews/final-review.md`

### 3. `phase-ship.md` 与真实生命周期不一致

**现象**

- 当前 lifecycle 已经明确分成 `phase-audit` 和 `phase-finalize`
- 但 playbook 文件仍然沿用 `phase-ship.md`

**风险**

- 文档、技能、doctor、自检的概念都被混淆
- 用户和维护者很难理解“终审”和“收尾”到底哪个阶段负责什么

**整改**

- 删除 `phase-ship.md`
- 拆成 `phase-audit.md` 与 `phase-finalize.md`

### 4. 仓库内命令优先读取已安装配置，导致本地源码验证结果失真

**现象**

- 直接运行仓库里的 `scripts/copilot-ilongrun`
- 但它会优先读取 `~/.copilot-ilongrun/config/model-policy.jsonc`
- 如果已安装目录还是旧配置，就会把旧角色映射和旧模型策略带回当前 run

**风险**

- 你以为自己在验证当前仓库代码
- 实际却混入了历史安装残留配置
- 容易出现“仓库已改、真实命令仍按旧逻辑跑”的假阳性

**整改**

- 仓库内命令改为优先读取当前仓库 `config/` 下的模型配置与 coding protocol
- 已安装命令继续读取 `~/.copilot-ilongrun/config/`
- 自检新增断言，禁止 `mission.modelAllocation` 再出现 `gpt54-audit-reviewer`

## P2 应修问题（已整改）

### 1. README 首页过于偏协议说明，不适合新手

**整改**

- README 重构为新手上手页
- 深层协议细节下沉到 `docs/快速开始.md` 和 `docs/架构与运行机制.md`

### 2. 模型配置说明存在版本漂移

**现象**

- `config/model-policy.jsonc` 头部说明仍停留在旧版本表达

**整改**

- 调整为“当前主线模型策略单一真值源”表述，避免版本误导

### 3. 当前文档职责边界不够清楚

**整改**

- `README.md`：新手入口
- `docs/快速开始.md`：操作展开版
- `docs/架构与运行机制.md`：机制解释与架构图
- `CHANGELOG.md` / 发版说明：版本历史

## P3 优化项（已整改）

### 1. 缺少一眼能看懂的架构图

**整改**

- README 增加简化总览图
- 架构文档增加详细生命周期图

### 2. 新手不知道运行后应该看哪些文件

**整改**

- README 与快速开始都增加“会生成哪些文件”章节
- 明确强调 `scheduler.json`、`reviews/final-review.md`、`COMPLETION.md`

## 审计结论

本轮整改后，项目在以下方面已经收敛：

- 当前命名与真实生命周期一致
- `--model` 语义与实现一致
- 终审路径与 agent 命名不再绑定某个具体模型
- README 更适合技术新手首次使用
- 文档层级职责更清晰

仍建议后续保持两条纪律：

1. 任何新 public path / public name 一旦落地，就不要再把模型名写进文件名
2. 任何阶段名变动，必须同步更新 doctor、自检、模板、README、架构图
