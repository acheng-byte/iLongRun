# 项目全局审计与整改说明

## 1. 背景

这份说明记录的是 iLongRun 在一次真实 coding 样本 run 复盘后，沉淀出的**通用整改原则**。

目标不是为了让某一个样例“看起来通过”，而是为了让任意 coding 场景下的 run 都满足以下标准：

- 生命周期真实可验证
- 完成态不可伪造
- review / audit / finalize 不可跳过
- verifier verdict 与 completion score 语义一致
- 不依赖真实 Copilot CLI 长跑，也能完成核心回归验证

## 2. 单一真值原则

### 2.1 run 完成态只认一个字段

从本轮整改开始：

- `scheduler.json` 的 **top-level `state`** 是 run 完成态唯一真值
- top-level `status` 只保留 legacy 兼容读取能力
- 一旦 `status` 与 `state` 冲突，verify / doctor 直接判定为 drift

### 2.2 ledger 写盘必须走 helper

完成态、verification、projection、active pointer 的更新必须统一经过 helper / ledger API：

- 禁止直接手写 top-level 状态字段冒充完成
- 新的 ledger sync 会自动清理 top-level `status`
- 任何完成态都必须可由 `scheduler.json + workstreams/*/status.json + completion artifacts` 复核

## 3. Gate Enforcement 原则

### 3.1 finalize 的必备条件

一个 coding run 进入真正完成态前，至少必须满足：

- `claimVerification` 已收敛
- phase guards 已收敛
- review gate 已完成
- `reviews/final-review.md` 存在且可解析
- `reviews/adjudication.md` 存在
- `COMPLETION.md` 成功生成
- `active-run-id` 已清理

任一条件缺失，都不应被视为真正 complete / completed。

### 3.2 review 伪完成阻断

如果某个 review workstream：

- `status.json` 标 done/complete
- 但 `result.md` / `evidence.md` 为空、pending、placeholder、TODO/TBD、模板残留

则该 workstream 必须被视为**伪完成**，不能继续推进到 audit / finalize。

### 3.3 audit 不能覆盖上游事实

final review / adjudication 的职责是汇总、裁决、降级或返回修复，不能直接覆盖掉上游 review 已经明确写出的“证据不足 / 仅 smoke / 未闭环”事实。

## 4. verifier 与评分语义

### 4.1 `ok=false` 时不得给乐观交付结论

如果 verify 结果存在：

- hard failure
- state drift
- finalize 关键产物缺失
- review 伪完成

则 completion score 必须降级或封顶，`deliveryVerdict` 也必须同步降级。

不允许再出现：

- `ok=false`
- 同时 `grade=A`
- 同时 `deliveryVerdict=prototype-ready`

### 4.2 delivery score 的定位

completion score 仍然保留“代码存在 / 接入主链 / 测试 / 运行态验证”四层结构，但它现在必须服从 gate 结论：

- hard failure → 分数封顶且 verdict=`blocked`
- drift only → 分数降级且 verdict=`state-drift`

## 5. sessionEnd 与收尾策略

### 5.1 sessionEnd 只做 precheck，不代替 finalize

`sessionEnd` 事件链路会自动做一次本地 precheck，并把：

- verification state
- recommended action
- failure class
- hook precheck 时间戳

写回账本。

但它**不会**凭空把 run 自动 finalize 成 complete。

### 5.2 completion 缺失时不得清 active

如果 run 被标成 complete/completed，但 `COMPLETION.md` 仍缺失，则：

- 不能自动清理 `active-run-id`
- 必须保留 active pointer，并在 verify / doctor 中明确报错

## 6. workspace 与运行态安全

### 6.1 工作区污染

verify / doctor 会检查当前 git 工作区是否错误跟踪了以下生成态目录：

- `.copilot-ilongrun/`
- `node_modules/`
- `dist/`
- `build/`
- `.next/`
- `coverage/`

如果这些目录被 git 跟踪，会被视为 run 卫生问题或交付污染问题。

### 6.2 workspace isolation 可重评估

workspace isolation 不是“prepare 时看一次就结束”。

如果 run 中途发生：

- `git init`
- tracked files 爆炸
- 生成态目录被纳入索引

后续 reconcile / verify 需要重新按**当前运行事实**评估，而不是继续相信初始化时的状态。

## 7. 回归与验证策略

本轮修复默认使用**非 Copilot CLI** 回归：

- `python3 scripts/selftest_ilongrun.py`
- `python3 scripts/lint_ilongrun_skills.py`
- `bash scripts/ilongrun-doctor --refresh-model-cache`
- 定向 helper 测试（verify / finalize / hook_event / ledger sync）
- 自造 run fixture 回归状态冲突、placeholder、缺 completion、工作区污染等场景

只有用户明确允许、且确有必要时，才考虑再做真实 Copilot 长跑验证。
