# iLongRun 架构与运行机制

## 1. 核心定位

iLongRun 不是单纯的 prompt 模板，而是围绕 GitHub Copilot CLI 构建的 **Planner-of-Planners 蜂群编排内核**。

从 v0.7.0 开始，coding 任务的定位进一步明确：

> **iLongRun 负责长运行账本与编排；`ilongrun-coding` 负责把 coding 任务约束成带方法学硬门禁的 Coding Discipline Kernel。**

## 2. 角色分工

- `Mission Governor`：总编排、裁决、重规划、收尾
- `Strategy Synthesizer`：生成策略、依赖边界、模式选择理由
- `Phase Planner`：拆 phase / wave
- `Workstream Planner`：把 worker 合同写成 mini-PRD
- `Ledger Syncer`：把 scheduler 真值回写到 plan / task-list 投影
- `Executor`：执行单个 workstream
- `Recovery Agent`：失败恢复与最小路径修正
- `Code Reviewer`：代码维度 review
- `Test Engineer`：测试证据 review
- `Security Auditor`：安全 review
- `Final Audit Reviewer`：最终终审与 release blocker 判断

## 3. 真值与投影

### 真值

- `scheduler.json`
- `workstreams/*/status.json`

### 投影

- `mission.md`
- `strategy.md`
- `plan.md`
- `task-list-N.md`
- `workstreams/*/brief.md`
- `reviews/*`

### 核心原则

- Markdown 是给人看的
- JSON 是系统账本
- `task-list` 勾选由结构化 checklist 回写
- workstream 声称完成但缺 result/evidence/status 闭环时，必须标记 `drift`
- coding verify 会额外生成 `reviews/delivery-audit.md`

## 5. coding 生命周期

新的 coding run 固定采用：

```text
phase-define
→ phase-plan
→ phase-build
→ phase-verify
→ phase-review
→ phase-audit
→ phase-finalize
```

### 各阶段职责

- `phase-define`
  - 锁定目标、边界、假设、成功标准、语言画像
- `phase-plan`
  - 生成 dependency graph、wave 划分、worker 合同
- `phase-build`
  - 推进 build slices 与 integration wave
  - build 前先做 workspace isolation assessment
  - 每个 build workstream 都要走 microcycle
- `phase-verify`
  - 固定测试、构建、接线、运行证据
  - claim verification 在这里开始收敛
- `phase-review`
  - 执行 code / test-evidence / security 三类专项评审
- `phase-audit`
  - 汇总 review 结果，做最终终审与 release blocker 判断
- `phase-finalize`
  - 只有 claim verification 完整时才允许完成态收尾

## 6. Swarm 编排语义

iLongRun 内部把 swarm 分成三类：

- `serial`
  - 必须串行推进，例如 finalize、release、终审
- `swarm-wave`
  - 有共同上游但彼此可并行的波次
- `super-swarm`
  - 完全独立、可大并行的工作流

对于 coding：

- 默认主模式是 `swarm-wave`
- build slice 可以并行
- review / audit / finalize 一律 internal
- 蜂群骨架不变，变的是每个 build worker 内部执行纪律更严格

## 7. `/fleet` 在架构里的位置

`/fleet` 只是 wave 的一个执行后端，不是真值层。

### 使用原则

- 只在当前 wave 合法时开启
- coding 场景只允许 `phase-build` 评估 `/fleet`
- 写集冲突或需要人工裁决时必须降级

### 运行态证据

- `runtime.fleetCapability`
- `runtime.fleetDispatch`
  - `completedWaves`
  - `degradedWaves`
  - `dispatchEvents[]`

verify / status 都会读取这些字段，而不是只看 mode 文案。

## 8. 方法学硬门禁

### workspace isolation

- 在 build 前先判断是否需要 worktree / branch / in-place
- 非 git 工作区允许 `skipped`，但必须给出原因

### task microcycle

固定顺序：

```text
spec-lock → red → verify-red → green → verify-green → self-review → spec-review → quality-review → handoff
```

### claim verification

- required build / verify workstream 缺 fresh evidence 时，禁止 finalize
- 只认新鲜命令证据，不认“我应该已经跑过了”

### root cause before fix

- failed / blocked workstream 先写 `rootCauseRecord`
- 再允许 minimal fix
- 再允许回到主流程

## 9. 独立 review gate

`phase-review` 是独立 gate，不允许被 `phase-audit` 替代。

默认 review matrix：

- `review-code`
- `review-test-evidence`
- `review-security`

只有这些 gate 收敛完成后，系统才会进入最终终审与 finalize。

## 10. vendorized method sources 的作用

仓库保留：

- `vendor/agent-skills/`

目的不是运行时直接依赖上游，而是：

- 固化来源
- 保留 LICENSE
- 保留方法学快照
- 再由 iLongRun 自己的 playbooks 做深度适配

## 11. 状态看板与 doctor

### `ilongrun-status`

会重点展示：

- 当前 run 状态 / phase / mode / profile / model
- completion score / delivery verdict
- phase / wave / workstream 进度
- coding protocol / swarm / review gate 状态
- methodology guard：workspace isolation / taskMicrocycle / claimVerification / rootCauseBeforeFix
- ledger sync / projection sync / active-run 指针
- fleet runtime 摘要

### `ilongrun-doctor`

会重点检查：

- launchers 是否完整
- Copilot 登录与模型缓存
- `/fleet` 探测能力
- coding protocol bundle 是否完整
  - `coding-protocol.jsonc`
  - `vendor/agent-skills/`
  - `skills/ilongrun-coding/*`
  - 专项 review agents
  - `lint_ilongrun_skills.py`

## 12. completion / audit 报告骨架

为保证人类可读性与机器可解析性，coding run 统一采用：

- Final Review：`Run Metadata → Summary → Findings → Suggested Fixes → Residual Risks → Verdict`
- Adjudication：`Run Metadata → Summary → Findings Intake → Decision → Next Actions → Verdict`
- Completion：`Run Metadata → Summary → Completion Score → Deliverables → Verification Evidence → Blockers → Verdict`

其中：

- 空节统一写 `- None.`
- `Verdict` 必须独立成节
- `Must-Fix / Should-Fix / Residual Risks` 标题保持稳定，供 verifier 解析
