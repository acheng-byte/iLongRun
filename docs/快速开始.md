# iLongRun 快速开始

## 1. 一键安装

```bash
curl -fsSL https://raw.githubusercontent.com/izscc/iLongRun/main/install.sh | bash
```

## 2. 检查环境

```bash
command -v ilongrun
command -v ilongrun-coding
ilongrun-doctor --refresh-model-cache
ilongrun-doctor --notify-test
```

> 一键安装现在会先执行彻底清理：卸载旧 `ilongrun / longrun / copilot-mission-control` 定义，删除旧缓存、旧命令、旧 skills / agents、`~/.copilot-ilongrun` 与 `~/.copilot-mission-control`，再安装新版插件与命令。
> 如需保留自定义 `~/.copilot-ilongrun/config/model-policy.jsonc`，请先自行备份。
> `--notify-test` 仅在 macOS 上有意义，用来检查系统提醒链路是否正常。

安装结束后，终端会直接给你一张中文安装看板，默认保持简洁无额外配色的原始风格与右侧开口框，顺手告诉你下一步该先跑哪个命令。
如果你想修改默认模型，看板也会直接提示配置文件位置和该改的字段。
而 `ilongrun-doctor --refresh-model-cache` 也会输出一张中文体检看板，把命令入口、登录状态、模型缓存、自检结果和 `/fleet` 能力一次讲清楚。

## 3. 最常用的几个命令

```bash
ilongrun "修复登录流程并补充测试"
ilongrun-coding "实现登录模块并补测试，最后做最终终审"
ilongrun-prompt "调研 3 个 AI Agent 编排方案并输出中文结论"
ilongrun-status latest
ilongrun-resume latest
```

其中：
- `ilongrun`：通用入口，自动推断 profile
- `ilongrun-coding`：显式 coding 入口，强制 `profile=coding`

## 4. 运行目录在哪里

每次运行都会在当前工作区生成唯一状态根目录：

```text
.copilot-ilongrun/
```

v0.4.0 之后：

- run 只允许存在于 `.copilot-ilongrun/runs/<run-id>/`
- 旧 `.copilot-ilongrun/<run-id>/` 会在 reconcile / verify / finalize 时自动收敛
- 旧 `.copilot-mission-control/` 会被归档到 `.copilot-ilongrun/legacy-imports/` 后删除

其中最重要的是：
- `mission.md`
- `strategy.md`
- `plan.md`
- `scheduler.json`
- `projection-sync.jsonl`
- `task-list-N.md`
- `workstreams/ws-*/`
- `reviews/gpt54-final-review.md`（coding 任务，文件名保持兼容）
- `reviews/adjudication.md`（coding 任务）

## 5. 默认模型在哪里改

主配置文件：

```bash
~/.copilot-ilongrun/config/model-policy.jsonc
```

默认规则：

- `ilongrun` / `status` / `prompt` / `resume` / `doctor` → `claude-sonnet-4.6`
- `ilongrun-coding` → `claude-opus-4.6`
- coding 最终终审 → `gpt-5.4`

优先级：

```text
--model > commandDefaults > skillDefaults > roleModels > fallback
```

## 6. 什么时候会用到 `/fleet`

只有满足下面条件才会自动尝试：
- 2 到 4 个相互独立的 workstream
- 写集不冲突或已显式分区
- 不依赖中途人工裁决
- 失败后可独立重试

若本机 Copilot CLI 不支持 `/fleet`，iLongRun 会自动降级为 `internal`。

从 v0.5.0 开始，`/fleet` 的 probe / dispatch / fallback 证据也会写进 runtime 账本；而每次 plan / task-list 投影回写时，会追加到 `projection-sync.jsonl`，便于 `status` 与复盘直接查看最近一次同步做了什么。
