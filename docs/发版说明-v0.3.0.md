# 发版说明 v0.3.0 — macOS 通知链路回归（含 `ilongrun-coding` 热修）

> 发版日期：2026-04-07（刷新）

## 概述

v0.3.0 先补回了 macOS 通知链路，随后又补了一次热修，把首发版本里遗漏的 `ilongrun-coding` 命令入口、doctor 检查和 README/快速开始同步补齐。

这一版的目标很明确：**让 iLongRun 在后台长跑时主动把状态提醒送到 Mac 用户面前，同时保证 coding 用户入口、安装结果与文档说明保持一致。**

## 这次改了什么

### 完整通知链路回归

现在 iLongRun 支持以下通知事件：
- `launched`
- `resumed`
- `recovery`
- `attention`
- `complete`
- `blocked`
- `checkpoint`

其中：
- detached 启动成功后会提醒
- resume 成功后会提醒
- 首次自动恢复 / fallback 时会提醒
- complete precheck 失败时会提醒
- 最终完成 / 阻塞时会提醒

### `ilongrun-coding` 命令热修

首发 v0.3.0 的问题是：
- `skills/ilongrun-coding` 已安装
- 但 shell launcher `ilongrun-coding` 没有被安装脚本补进去
- README / 快速开始 / doctor 也没有把这个缺口暴露清楚

本次刷新后：
- 一键安装会补齐 `ilongrun-coding`
- `copilot-ilongrun coding ...` 成为正式高级入口
- `ilongrun-doctor` 会检查 launcher 完整性，并显式报告 coding 入口是否缺失
- `ilongrun-coding` 会显式强制 `profile=coding`

这意味着：
- `ilongrun` 仍然是通用入口，自动推断 profile
- `ilongrun-coding` 是显式 coding 入口，适合你已经明确知道这是代码任务的场景

### 通知后端：增强优先，基础保底

通知后端按以下顺序工作：
1. `terminal-notifier`
2. `osascript`
3. 非 macOS 或 dry-run 时输出模拟结果

这意味着：
- 装了 `terminal-notifier` 时，会优先获得更完整的通知体验
- 没装时，macOS 仍然可以通过系统自带 `osascript` 收到基础提醒

### 点击通知后的打开目标

这次**没有照搬 LongRun 的独立“启动看板”文件**，而是直接复用 iLongRun 已有产物：
- `launched` / `resumed` / `recovery` / `attention`：优先打开 `plan.md`
- `complete` / `blocked`：优先打开 `COMPLETION.md`

这样可以减少额外文件协议，更贴合 iLongRun 当前结构。

### doctor 自检命令

现在可以直接用下面命令检查提醒链路：

```bash
ilongrun-doctor --notify-test
```

如果没有看到提醒横幅，请优先检查：
- 系统设置 > 通知
- 专注模式
- 通知摘要

此外，`ilongrun-doctor` 现在还会检查以下命令是否完整安装：
- `ilongrun`
- `ilongrun-coding`
- `ilongrun-prompt`
- `ilongrun-resume`
- `ilongrun-status`
- `ilongrun-doctor`
- `copilot-ilongrun`

## 安装 / 升级

```bash
curl -fsSL https://raw.githubusercontent.com/izscc/iLongRun/main/install.sh | bash
```

安装后建议执行：

```bash
command -v ilongrun-coding
ilongrun-doctor --refresh-model-cache
ilongrun-doctor --notify-test
```

## 兼容性

- 不改变 iLongRun 的 run 目录协议
- 不改变 `scheduler.json` 结构
- 新增 `ilongrun-coding` 命令入口，其他既有入口保持不变
- 与 v0.2.1 保持兼容

## 说明

- 可通过 `ILONGRUN_NOTIFICATIONS=0` 关闭通知
- 卸载 helper 时不会自动删除第三方 `terminal-notifier` 安装物，这与 LongRun 行为保持一致
- `skills/ilongrun-coding` 仍然是内部 discipline skill；终端用户应使用 `ilongrun-coding` 命令，而不是把它当成顶层 skill 入口
