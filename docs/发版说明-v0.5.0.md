# 发版说明 v0.5.0 — 真完成度审计、账本同步闭环、`/fleet` 证据化

> 发版日期：2026-04-07

## 概述

v0.5.0 是一次围绕 **“长跑是否真的交付了”** 的系统性补强，直接来源于这次对 `test4-game / Vibe Arena` 的深度复盘。

这一版主要解决三类问题：

1. **账面完成 ≠ 真实交付**：以前只看 scheduler/review/result，很容易把“代码很多、测试也过了，但核心主链没接上”的项目误判成完成。
2. **真值与投影容易漂移**：`plan.md / task-list-N.md` 不会动态反映真实 workstream 状态，甚至 finalize 后 `active-run-id` 还会残留。
3. **`/fleet` 只有账面字段，没有运行证据**：看得到 mode，却看不到 probe / dispatch / degraded / completed 的可信证据链。

这一版的目标就是：**让 iLongRun 不只是“能长跑”，而是能对自己的交付真实性、账本一致性和运行稳定性负责。**

## 这次改了什么

### 1) 账本协议与 finalize/verifier 契约补强

重点修复了 finalize / verify / hook / reconcile 之间的核心契约问题：

- 统一识别 `complete / completed / finalized`
- `finalize` 后自动清理 `active-run-id`
- 已完成 run 不再继续吸附后续 hook 事件
- verifier 不再硬依赖 `workstreams[*].index`
- `scheduler.taskLists[] / taskListPath / checklist status` 正式进入账本协议
- reconcile / sync 改为 **先补投影再 verify**

这意味着：

- 历史 run 缺字段时不再一上来 traceback
- drift 会被保留为 drift finding，而不是直接把 finalize 打崩
- 真值链现在真正收敛到：

```text
scheduler.json + workstreams/*/status.json
```

### 2) task-list / plan 动态同步真正落地

以前的 `task-list-N.md` 更像静态模板；这版之后：

- `task-list` 的 `[x] / [ ]` 由 checklist 状态确定性回写
- workstream 声称完成但缺 result/evidence/status 闭环时，显式标记 `⚠ drift`
- coding run 会自动生成 `reviews/adjudication.md`
- `Ledger Syncer` 角色正式补上，默认采用 **确定性脚本优先，子代理兜底**

同时新增：

```text
projection-sync.jsonl
```

每次账本同步都会留下结构化投影日志，记录：

- sync reason
- verification / drift 数量
- task-list 数量
- 最近改写了哪些投影文件

这让“为什么 plan/task-list 被改了”第一次具备了显式审计线索。

### 3) 新增“假完成检测器”

为了解决 Vibe Arena 这类“模块存在，但没真正接入主链”的假完成问题，这一版新增：

- `scripts/_ilongrun_delivery_audit.py`
- `scripts/scan_ilongrun_delivery_gaps.py`

它会扫描 JS/TS import graph，并识别：

- 未接主链模块
- 重名核心模块（例如两套 `RoomManager`）
- 明显的 placeholder / noop provider

对 coding run：

- verify 会自动生成 `reviews/delivery-audit.md`
- 高置信度结果会升级为 drift finding
- 可直接阻断账面完工

### 4) 新增真实完成度评分

verify / finalize / status board 现在都会给出 evidence-based completion score：

- `codeExists`
- `wiredIntoEntry`
- `tested`
- `runtimeValidated`

并汇总成：

- `overall`
- `grade`
- `deliveryVerdict`

它不是为了取代人工判断，而是为了防止系统再次把：

> “高完成度原型代码库”

误判成：

> “真正可交付的项目”

### 5) 状态看板升级为本地确定性看板

`ilongrun-status` 现在优先走本地 helper，而不是依赖模型临时组织输出。

新看板统一展示：

- completion score / delivery verdict
- phase / wave / workstream 进度
- verification / drift / risk 摘要
- ledger sync / projection sync / active-run 指针
- `/fleet` runtime 证据

风格保持与 install / doctor / detached launch 一致：

- 中文
- 简洁
- 右侧开口框
- 不依赖额外配色才能看懂

### 6) `/fleet` 从“账面可用”升级到“有证据可审计”

这一版新增 `/fleet` 运行证据链：

- `runtime.fleetCapability`
  - `status / reason / checkedAt`
  - `probeModel / probeModelDisplay`
  - `cache / source / checkedBy`
  - `rawOutputDigest`
- `runtime.fleetDispatch`
  - `completedWaves / degradedWaves`
  - `lastDispatchedWave / lastOutcome / lastOutcomeAt`
  - `dispatchEvents[]`

verify 会检查：

- 是否真的做过 probe
- 是否缺 dispatch evidence
- completed / degraded wave 是否引用了不存在的 wave
- 事件是否缺 `waveId / outcome / observedAt`

这样 `/fleet` 不再只是一个 mode 字段，而是可复盘的运行证据链。

### 7) review / adjudication / completion 模板统一

coding 生命周期里的三类报告现在统一成固定骨架：

- Final Review  
  `Run Metadata → Summary → Findings → Suggested Fixes → Residual Risks → Verdict`
- Adjudication  
  `Run Metadata → Summary → Findings Intake → Decision → Next Actions → Verdict`
- Completion  
  `Run Metadata → Summary → Completion Score → Deliverables → Verification Evidence → Blockers → Verdict`

同时修掉了一个隐藏问题：

- `## Verdict` 不再被误解析到 `Residual Risks / Defer`

这能显著降低“标题自由发挥导致账本误读”的风险。

## 升级建议

```bash
curl -fsSL https://raw.githubusercontent.com/izscc/iLongRun/main/install.sh | bash
ilongrun-doctor --refresh-model-cache
```

如果你正在排查：

- task-list 为什么一直不勾选
- 某次 run 为什么账面 complete 但状态板不认
- `/fleet` 到底有没有真的 dispatch 过
- 项目是不是“文件很多但没真正交付”

建议先对目标工作区执行：

```bash
ilongrun-status latest
```

必要时再对历史 run 执行：

```bash
python3 scripts/sync_ilongrun_ledger.py --workspace <workspace> --run-id <run-id> --clean-active-on-complete --print-json
```

## 这一版的核心结论

v0.5.0 不是“多加几个文件”的功能更新，而是一次针对 iLongRun 内核可信度的补课：

> **让 iLongRun 不再只会记录“做了很多事”，而是开始认真判断“这件事到底是不是真的做成了”。**
